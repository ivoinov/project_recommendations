  <action>
Add a publish service that:
- Uses advisory locks per shop schema to avoid concurrent runs.
- Checks for same-day runs and skips if a run exists (success or failed).
- Compares product/order counts to previous run; skip publish if counts unchanged.
- Computes raw candidates per seed SKU for placements "pdp" and "cart" using co-purchase counts and content-based candidates (no in-stock/exclusion filtering here).
- Stores candidates by candidate_type ("co_purchase", "content_based", and a "merged" list) so API responses can preserve the existing fields.
- Assigns a numeric score per candidate and sorts by score desc with popularity counts as tie-break, then SKU.
- Inserts candidates with new version, updates publish state in a single transaction, and retains only last 2 versions.
- After a successful publish, invalidates cache keys for the shop/placement and warms cache for the newest version.
- On error after version increment, rolls back to previous version and records run status as failed.

Implement helper changes:
- In baseline_recommendation_service.py, add a function (e.g., build_precompute_candidates) that returns raw candidate lists without applying filter_candidates.
- In order_repository.py add count_orders() and get_sku_popularity(skus) helpers.
- In product_repository.py add count_products() and list_all_skus() helpers.
- In shop_db.py add list_shop_ids(db) to read schemas matching "shop_%" from information_schema and normalize back to shop_id.
  </action>
